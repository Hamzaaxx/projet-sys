What is canaryfs?

canaryfs is a cybersecurity tool that plants fake sensitive files (called canaries or honeypot files) inside a directory, then silently watches them using the Linux kernel's inotify subsystem. The moment any user, process, or malware touches one of these files, canaryfs captures a full forensic snapshot and fires an alert.

This technique is used in real enterprise environments under the names canary tokens or honeypot files.

The 3 Execution Modes

-f Fork

Each canary file is watched by its own child process created with ( ) &. The parent process waits for all children with wait.

-t Thread

A compiled C program (monitor_thread) uses real POSIX pthreads. Each thread watches one canary file. All threads share the same process memory.

-s Subshell

All watchers run inside a detached background subshell ( ... ) &. The terminal is immediately free — canaryfs runs as a daemon.

default (no flag)

A single inotifywait watches the entire directory recursively. Only fires for files that are in the canary registry.

How It Works

Full execution flow from command to alert

Step-by-step flow

  1. User runs the command
    ./canaryfs -t -p 5 /tmp/honeypot
    main() is called with all arguments.
  2. parse_args() — read flags
    getopts loops through flags. Sets EXEC_MODE=thread, CANARY_COUNT=5, TARGET_DIR=/tmp/honeypot.
  3. init_log_dir() — prepare logging
    Creates /var/log/canaryfs/ if it doesn't exist. Opens history.log.
  4. validate() — safety checks
    Verifies inotifywait is installed, a target directory was given, and it actually exists.
  5. compile_thread_monitor() — build C binary (thread mode only)
    If lib/monitor_thread doesn't exist yet, it runs gcc to compile monitor_thread.c.
  6. plant_canaries() — create bait files
    Writes 5 fake files into /tmp/honeypot. Records each path in /tmp/.canaryfs_registry.
  7. run_thread() — start watchers
    Reads all paths from the registry and passes them to the C binary: monitor_thread history.log file1 file2 ...
  8. watch_file() — inotify loop (inside C thread)
    Each thread runs inotifywait -m on its file. Blocks waiting for kernel events.
  9. capture_forensics() — who did it?
    On event: runs lsof to find the process accessing the file, reads /proc/<pid>/status for the parent PID.
  10. log_alert() + fire_alert() — report
    Writes the ALERT line to the terminal and log file. If -a was set, also writes to syslog and sends a desktop notification.

Data Flow Diagram

Kernel Interface: inotify

inotify is a Linux kernel subsystem that lets programs subscribe to filesystem events. Instead of checking a file repeatedly (polling), the kernel notifies the program the instant something happens. This makes canaryfs efficient — it uses zero CPU while waiting.

EventTriggered when
accessFile was read
openFile was opened
modifyFile contents were changed

Forensic Capture: lsof + /proc

When inotify fires, canaryfs immediately runs two OS queries to identify the attacker:

lsof <file>

Lists all processes that currently have the file open. Returns the process name, PID, and UID of whoever is accessing the canary.

/proc/<pid>/status

The Linux /proc filesystem exposes live process information. The PPid field reveals which process launched the attacker's process (the parent).

Argument Parsing — parse_args()

parse_args() {
    local opt
    while getopts ":hftsal:rp:" opt; do
        case "${opt}" in
            h) show_help; exit 0 ;;
            f) EXEC_MODE="fork" ;;
            t) EXEC_MODE="thread" ;;
            s) EXEC_MODE="subshell" ;;
            a) ALERT_MODE=true ;;
            l) LOG_DIR="${OPTARG}"; LOG_FILE="${LOG_DIR}/history.log" ;;
            r) run_restore; exit $? ;;
            p) CANARY_COUNT="${OPTARG}" ;;
            :) log_error "Option -${OPTARG} requires an argument"; show_help; exit 101 ;;
            ?) log_error "Unknown option: -${OPTARG}";             show_help; exit 100 ;;
        esac
    done
    shift $((OPTIND - 1))
    TARGET_DIR="$1"
}

How getopts works

The string ":hftsal:rp:" tells getopts which flags are valid:

  • Leading : — silent mode: handle errors ourselves (the ? and : cases)
  • A letter alone (e.g. f) — flag with no argument
  • A letter followed by : (e.g. l:) — flag that requires an argument, stored in $OPTARG

After the loop, shift $((OPTIND - 1)) removes all processed flags from $@, leaving only the target directory as $1.

Validation — validate()

validate() {
    if ! command -v inotifywait &>/dev/null; then
        log_error "inotifywait not found — install inotify-tools"
        show_help; exit 102
    fi
    if [[ -z "${TARGET_DIR}" ]]; then
        log_error "Missing mandatory parameter: target_directory"
        show_help; exit 101
    fi
    if [[ ! -d "${TARGET_DIR}" ]]; then
        log_error "Target directory does not exist: ${TARGET_DIR}"
        show_help; exit 103
    fi
}

Three checks in order: tool available → argument given → directory exists. Each failure prints the help and exits with a specific code.

The Four Execution Modes

run_fork() {
    log_info "Mode: fork — one child process per canary"
    while IFS= read -r canary_file; do
        ( monitor_file "${canary_file}" ) &   # ( ) creates a subshell = new child process
    done < "${CANARY_REGISTRY}"
    wait   # parent blocks here until ALL children exit
    log_info "All fork watchers exited"
}

run_thread() {
    log_info "Mode: thread — POSIX pthreads via C"
    mapfile -t files < "${CANARY_REGISTRY}"  # read registry into array
    "${MONITOR_THREAD_BIN}" "${LOG_FILE}" "${files[@]}"  # pass all files to C binary
    log_info "All thread watchers exited"
}

run_subshell() {
    log_info "Mode: subshell — background daemon"
    (                                          # open a subshell
        while IFS= read -r canary_file; do
            monitor_file "${canary_file}" &
        done < "${CANARY_REGISTRY}"
        wait
    ) &                                        # detach the whole subshell to background
    log_info "Daemon PID: $!"                  # $! = PID of the last background process
}

run_default() {
    log_info "Mode: default — sequential directory watch"
    monitor_directory "${TARGET_DIR}"          # one inotifywait for the whole dir
}
ModeMechanismParallel?Blocking?
fork( ) & per file + waitYesYes — parent waits
threadC pthread_createYesYes — joins all threads
subshell( ) & groupYesNo — returns immediately

Auto-Compile — compile_thread_monitor()

compile_thread_monitor() {
    [[ -x "${MONITOR_THREAD_BIN}" ]] && return   # already compiled, skip
    if ! command -v gcc &>/dev/null; then
        log_error "gcc not found — install build-essential"
        exit 1
    fi
    log_info "Compiling monitor_thread.c ..."
    gcc -o "${MONITOR_THREAD_BIN}" "${SCRIPT_DIR}/lib/monitor_thread.c" -lpthread -Wall -O2 \
        || { log_error "Compilation failed"; exit 1; }
    log_info "Compiled: ${MONITOR_THREAD_BIN}"
}

This function only runs when -t is used. It checks if the binary already exists (-x = exists and is executable). If not, it compiles it automatically. The -lpthread flag links the POSIX threads library.

Entry Point — main()

lib/plant.sh

Creates fake bait files in the target directory and registers them

Full Source

lib/plant.sh
#!/bin/bash
CANARY_TYPES=("id_rsa" ".env" "credentials.json" "shadow.bak" "backup.sql")

plant_canaries() {
    local target_dir="$1"
    local count="$2"

    > "${CANARY_REGISTRY}"    # empty the registry file
    log_info "Planting ${count} canary files in ${target_dir}"

    for (( i=0; i<count; i++ )); do
        local type="${CANARY_TYPES[$((i % ${#CANARY_TYPES[@]}))]}"
        local dest="${target_dir}/${type}"
        _write_canary "${type}" "${dest}"
        echo "${dest}" >> "${CANARY_REGISTRY}"
        log_info "Planted: ${dest}"
    done

    log_info "Done — ${count} canary files planted"
}

_write_canary() {
    local type="$1"
    local dest="$2"
    local template="${SCRIPT_DIR}/canaries/${type}.tpl"

    if [[ -f "${template}" ]]; then
        cp "${template}" "${dest}"   # use custom template if it exists
    else
        case "${type}" in
            id_rsa) cat > "${dest}" <<'EOF'
-----BEGIN OPENSSH PRIVATE KEY-----
... fake key data ...
-----END OPENSSH PRIVATE KEY-----
EOF
            ;;
            .env) cat > "${dest}" <<'EOF'
APP_ENV=production
DB_PASS=Sup3rS3cr3tP@ssw0rd!
AWS_ACCESS_KEY_ID=AKIAIOSFODNN7FAKE123
EOF
            ;;
            # ... other types ...
        esac
    fi

    chmod 600 "${dest}"    # make it look like a real secret file
}

How the file type cycle works

The expression i % ${#CANARY_TYPES[@]} is a modulo operation:

  • ${#CANARY_TYPES[@]} = number of elements in the array = 5
  • When i=0: 0 % 5 = 0 → id_rsa
  • When i=1: 1 % 5 = 1 → .env
  • When i=5: 5 % 5 = 0 → id_rsa again

So if you plant 12 canaries, it cycles through all 5 types twice, then plants 2 more. This means each directory gets at most one of each type.

What is CANARY_REGISTRY? It is a plain text file at /tmp/.canaryfs_registry. Each line is the full path to one planted canary. The monitor uses this list to know which files to watch, and restore uses it to know which files to delete.

Why chmod 600?

chmod 600 sets permissions to read/write for owner only — no access for group or others. Real SSH keys and credential files always have this permission. Without it, the fake files would look suspicious to an attacker checking file metadata.

Template vs Inline

_write_canary first checks if a .tpl file exists in the canaries/ folder. This lets you customize the fake file content without editing the script. If no template exists, it falls back to the hardcoded inline content.

Full Source

lib/monitor.sh
#!/bin/bash

monitor_file() {
    local target_file="$1"

    inotifywait -m -e access,open,modify "${target_file}" 2>/dev/null \
    | while read -r dir event file; do
        _capture_forensics "${dir}${file:-$(basename "${target_file}")}" "${event}"
    done
}

monitor_directory() {
    local target_dir="$1"

    log_info "Watching directory: ${target_dir}"
    inotifywait -m -r -e access,open,modify \
        --format '%w%f %e %T' --timefmt '%Y-%m-%d-%H-%M-%S' \
        "${target_dir}" 2>/dev/null \
    | while read -r full_path event timestamp; do
        if grep -qxF "${full_path}" "${CANARY_REGISTRY}" 2>/dev/null; then
            _capture_forensics "${full_path}" "${event}"
        fi
    done
}

_capture_forensics() {
    local file="$1"
    local event="$2"

    local lsof_info proc_name pid uid ppid
    lsof_info=$(lsof "${file}" 2>/dev/null | tail -n +2 | head -1)
    proc_name=$(echo "${lsof_info}" | awk '{print $1}')
    pid=$(echo "${lsof_info}"       | awk '{print $2}')
    uid=$(echo "${lsof_info}"       | awk '{print $3}')

    if [[ -z "${pid}" ]]; then
        pid="unknown"; proc_name="unknown"; uid="$(id -u)"
    fi

    ppid=$(grep PPid "/proc/${pid}/status" 2>/dev/null | awk '{print $2}')
    ppid="${ppid:-unknown}"

    local msg="canary accessed — file=${file} event=${event} pid=${pid} process=${proc_name} uid=${uid} ppid=${ppid}"
    log_alert "${msg}"
    fire_alert "${msg}"
}

_capture_forensics() — step by step

  1. Run lsof on the accessed file
    lsof "${file}" lists all processes that have the file open right now.
    tail -n +2 skips the header line. head -1 takes only the first process.
  2. Extract process info with awk
    Column 1 = process name, Column 2 = PID, Column 3 = UID.
    Example lsof output: cat 4821 hamza 4r REG ...
  3. Fallback if lsof returns nothing
    If the process closed the file before lsof ran, pid/process are set to "unknown".
    uid falls back to id -u (the UID of whoever is running canaryfs).
  4. Read PPID from /proc
    /proc/<pid>/status is a live view of the process. The PPid: line gives the parent process ID — useful to trace back to the shell or script that launched the attacker's command.
  5. Build and send the alert message
    Calls log_alert() to write to the log file and fire_alert() for syslog/notify-send.

lib/monitor_thread.c

Real POSIX threads in C — one thread per canary file

Why C? Bash has no real threads. Background jobs (&) are separate processes, not threads. To satisfy the -t requirement with actual POSIX pthreads (shared memory, same process), we write a small C program and call it from Bash.

Full Source

lib/monitor_thread.c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <pthread.h>      // POSIX threads
#include <unistd.h>
#include <time.h>
#include <pwd.h>           // getpwuid — username from UID

// Each thread gets this struct as its argument
typedef struct {
    char file[512];       // path to the canary file this thread watches
    char log_file[512];   // path to history.log
} WatchArgs;

// Write one ALERT line to terminal and log file
static void write_alert(const char *log_file, const char *canary_file,
                        const char *event, const char *pid,
                        const char *process, const char *uid, const char *ppid) {
    time_t now = time(NULL);
    struct tm *t = localtime(&now);
    char timestamp[32];
    strftime(timestamp, sizeof(timestamp), "%Y-%m-%d-%H-%M-%S", t);

    const char *username = getpwuid(getuid()) ? getpwuid(getuid())->pw_name : "unknown";

    char line[2048];
    snprintf(line, sizeof(line),
        "%s : %s : ALERT : canary accessed — file=%s event=%s pid=%s process=%s uid=%s ppid=%s\n",
        timestamp, username, canary_file, event, pid, process, uid, ppid);

    fputs(line, stdout); fflush(stdout);   // print to terminal

    FILE *fp = fopen(log_file, "a");       // append to log file
    if (fp) { fputs(line, fp); fclose(fp); }
}

// Run lsof + read /proc to identify who accessed the file
static void capture_forensics(const char *canary_file, const char *event,
                               const char *log_file) {
    char cmd[1024], lsof_out[512] = {0};
    char proc_name[128] = "unknown", pid_str[32] = "unknown";
    char uid_str[32]    = "unknown", ppid_str[32] = "unknown";

    snprintf(cmd, sizeof(cmd), "lsof '%s' 2>/dev/null | awk 'NR==2'", canary_file);
    FILE *fp = popen(cmd, "r");
    if (fp) {
        if (fgets(lsof_out, sizeof(lsof_out), fp))
            sscanf(lsof_out, "%127s %31s %31s", proc_name, pid_str, uid_str);
        pclose(fp);
    }

    if (strcmp(pid_str, "unknown") != 0) {
        char proc_path[64];
        snprintf(proc_path, sizeof(proc_path), "/proc/%s/status", pid_str);
        FILE *ps = fopen(proc_path, "r");
        if (ps) {
            char buf[128];
            while (fgets(buf, sizeof(buf), ps))
                if (strncmp(buf, "PPid:", 5) == 0) {
                    sscanf(buf + 5, "%31s", ppid_str); break;
                }
            fclose(ps);
        }
    }

    write_alert(log_file, canary_file, event, pid_str, proc_name, uid_str, ppid_str);
}

// Thread function: each thread runs this, watching one file
void *watch_file(void *arg) {
    WatchArgs *w = (WatchArgs *)arg;

    char cmd[600];
    snprintf(cmd, sizeof(cmd),
        "inotifywait -m -e access,open,modify --format '%%e' '%s' 2>/dev/null",
        w->file);

    FILE *fp = popen(cmd, "r");   // open inotifywait as a stream
    if (!fp) { free(w); return NULL; }

    char event[64];
    while (fgets(event, sizeof(event), fp)) {   // block until event
        event[strcspn(event, "\n")] = '\0';     // strip newline
        capture_forensics(w->file, event, w->log_file);
    }

    pclose(fp); free(w); return NULL;
}

int main(int argc, char *argv[]) {
    // argv: monitor_thread <log_file> <file1> <file2> ...
    if (argc < 3) {
        fprintf(stderr, "Usage: monitor_thread <log_file> <file1> [file2 ...]\n");
        return 1;
    }

    char *log_file = argv[1];
    int n = argc - 2;   // number of canary files

    pthread_t *threads = malloc((size_t)n * sizeof(pthread_t));

    for (int i = 0; i < n; i++) {
        WatchArgs *w = malloc(sizeof(WatchArgs));
        strncpy(w->file,     argv[i + 2], 511);
        strncpy(w->log_file, log_file,    511);
        pthread_create(&threads[i], NULL, watch_file, w);
        //             ^           ^      ^            ^
        //        thread ID    attrs    function    argument
    }

    for (int i = 0; i < n; i++)
        pthread_join(threads[i], NULL);   // wait for each thread to finish

    free(threads);
    return 0;
}

Key Concepts

WatchArgs struct

When pthread_create launches a thread, it can only pass a single void* argument. We pack everything the thread needs (file path + log file path) into a WatchArgs struct, allocate it with malloc, and pass its pointer. Each thread receives its own copy.

pthread_create

pthread_create(&threads[i], NULL, watch_file, w);
ArgumentMeaning
&threads[i]Where to store the thread ID
NULLDefault thread attributes
watch_fileFunction the thread will run
wArgument passed to watch_file

pthread_join

After creating all threads, the main thread calls pthread_join for each one. This blocks the main thread until that thread exits. Without join, the program would exit immediately and kill all threads.

popen — running inotifywait as a stream

popen(cmd, "r") runs a shell command and returns a FILE* to read its output line by line. Here each thread runs inotifywait -m through popen, so every line printed by inotifywait is one event that triggers capture_forensics.

Thread vs Process: the key difference

fork mode (Bash &)

Each ( ) & creates a completely separate process. Separate memory, separate file descriptors. The kernel must copy the parent's memory space.

thread mode (pthreads)

All threads live in the same process. They share the same memory space and file descriptors. Lighter weight, faster creation, lower memory usage.

Verify it on Kali cat /proc/$(pgrep monitor_thread)/status | grep Threads
You will see Threads: N where N equals the number of canary files — all inside one single process.

#!/bin/bash

init_log_dir() {
    mkdir -p "${LOG_DIR}" || { echo "ERROR: cannot create log dir ${LOG_DIR}" >&2; exit 1; }
    touch "${LOG_FILE}"
}

_log() {
    local level="$1"
    local message="$2"
    echo "$(date +'%Y-%m-%d-%H-%M-%S') : $(whoami) : ${level} : ${message}" | tee -a "${LOG_FILE}"
}

log_info()  { _log "INFOS" "$1"; }
log_error() { _log "ERROR" "$1" >&2; }
log_alert() { _log "ALERT" "$1"; }

How _log() works

The core of the logging system is one line:

echo "..." | tee -a "${LOG_FILE}"

tee -a reads from stdin and writes to two places at once: standard output (the terminal) and the log file (in append mode -a). This means you always see alerts live on screen while they are also permanently saved.

Log levels

Why stderr for errors? log_error redirects to >&2 (stderr). This lets scripts that call canaryfs separate normal output from error messages, for example: ./canaryfs ... 2>/dev/null would hide errors but show info/alerts.

#!/bin/bash

fire_alert() {
    local message="$1"
    [[ "${ALERT_MODE}" != true ]] && return   # do nothing if -a was not set

    logger -t "canaryfs" -p auth.warning "${message}"   # write to syslog

    if command -v notify-send &>/dev/null; then          # if desktop notifs available
        notify-send --urgency=critical "canaryfs ALERT" "${message}"
    fi
}

Two alert channels

logger → syslog

logger is a standard Unix command that writes a message to the system log daemon. Flag -t canaryfs tags the message, -p auth.warning sets the facility and severity. The message ends up in /var/log/syslog or /var/log/auth.log.

notify-send → desktop popup

notify-send sends a notification to the desktop notification daemon (works on GNOME, KDE, etc.). --urgency=critical makes it appear in red with high priority. It is optional — checked with command -v before calling.

Why guard with ALERT_MODE?

Writing to syslog for every test run would pollute system logs. The -a flag makes it opt-in — you only enable it in production monitoring, not during development or testing.

Read syslog entries on Kali

tail -f /var/log/syslog | grep canaryfs
# or
journalctl -f -t canaryfs

lib/restore.sh

Removes all planted canary files — requires root privileges

Full Source

lib/restore.sh
#!/bin/bash

run_restore() {
    if [[ $EUID -ne 0 ]]; then           # EUID = effective user ID; 0 = root
        log_error "Option -r requires root privileges"
        show_help; exit 104
    fi

    if [[ ! -f "${CANARY_REGISTRY}" ]]; then
        log_info "No canary registry found — nothing to restore"
        return 0
    fi

    local count=0
    log_info "Removing all planted canaries ..."

    while IFS= read -r canary_file; do   # read registry line by line
        if [[ -f "${canary_file}" ]]; then
            rm -f "${canary_file}"
            log_info "Removed: ${canary_file}"
            (( count++ ))
        fi
    done < "${CANARY_REGISTRY}"

    rm -f "${CANARY_REGISTRY}"           # delete the registry itself
    log_info "Restore complete — ${count} canary files removed"
}

How it works

  1. Check for root
    $EUID is the Effective User ID. Root always has EUID 0. Any non-zero value means the script is not running as root, so it exits with code 104.
  2. Check registry exists
    If /tmp/.canaryfs_registry doesn't exist, there's nothing to clean up. Exit gracefully.
  3. Delete each canary
    Read the registry line by line. For each path, check the file still exists (-f) then remove it with rm -f. The -f on rm means no error if already gone.
  4. Delete the registry
    After all canaries are removed, the registry file itself is deleted so future runs start clean.
Why root only? Canary files may have been planted in system directories (like /etc) that require elevated permissions to delete. Also, restricting restore to root prevents a non-privileged attacker from simply removing the canaries after being detected.

Usage

sudo ./canaryfs -r /home/hamza

The -r flag is caught in parse_args() before any other processing and calls run_restore() immediately.

test_light.sh — Baseline

tests/test_light.sh
#!/bin/bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
TEST_DIR="/tmp/canaryfs_test_light"
RESULTS_FILE="/tmp/canaryfs_light_results.txt"

mkdir -p "${TEST_DIR}"
> "${RESULTS_FILE}"

echo "=== LIGHT TEST (5 canaries, 1 directory) ===" | tee -a "${RESULTS_FILE}"

for mode in fork thread subshell; do
    echo "--- Mode: ${mode} ---" | tee -a "${RESULTS_FILE}"

    START=$(date +%s%N)                       # nanosecond timestamp
    "${SCRIPT_DIR}/canaryfs" -${mode:0:1} -p 5 "${TEST_DIR}" &
    WATCHER_PID=$!

    sleep 1                                   # wait for watcher to start

    TRIGGER_START=$(date +%s%N)
    cat "${TEST_DIR}/.env" &>/dev/null        # trigger the canary
    TRIGGER_END=$(date +%s%N)

    sleep 1                                   # let detection fire
    kill "${WATCHER_PID}" 2>/dev/null
    END=$(date +%s%N)

    TOTAL_MS=$(( (END - START) / 1000000 ))
    TRIGGER_MS=$(( (TRIGGER_END - TRIGGER_START) / 1000000 ))

    echo "Total runtime : ${TOTAL_MS} ms"  | tee -a "${RESULTS_FILE}"
    echo "Trigger time  : ${TRIGGER_MS} ms" | tee -a "${RESULTS_FILE}"

    sudo "${SCRIPT_DIR}/canaryfs" -r "${TEST_DIR}"   # clean up
done

Key technique: nanosecond timing

date +%s%N returns Unix time in nanoseconds. Subtracting two values and dividing by 1,000,000 gives milliseconds. This is much more precise than date +%s (which only gives seconds).

The mode flag shortcut

-${mode:0:1} takes the first character of the mode string: fork → f, thread → t, subshell → s. So the loop automatically tests all three modes with the right flag.

test_medium.sh — Moderate Load

tests/test_medium.sh
#!/bin/bash
DIRS=()
for i in $(seq 1 5); do
    d="/tmp/canaryfs_test_medium_${i}"
    mkdir -p "${d}"
    DIRS+=("${d}")
done

for mode in fork thread subshell; do
    PIDS=()
    START=$(date +%s%N)

    for dir in "${DIRS[@]}"; do
        "${SCRIPT_DIR}/canaryfs" -${mode:0:1} -p 10 "${dir}" &   # 10 canaries per dir
        PIDS+=($!)
    done

    sleep 2

    for dir in "${DIRS[@]}"; do
        cat "${dir}/.env"    &>/dev/null &   # trigger in all dirs
        cat "${dir}/id_rsa"  &>/dev/null &   # simultaneously
    done
    wait

    sleep 1
    for pid in "${PIDS[@]}"; do kill "${pid}" 2>/dev/null; done
    END=$(date +%s%N)

    echo "Total runtime : $(( (END - START) / 1000000 )) ms"
done

test_heavy.sh — Stress Test

tests/test_heavy.sh
#!/bin/bash
BASE="/tmp/canaryfs_test_heavy"
DIRS=()
for i in $(seq 1 10); do
    d="${BASE}/dir_${i}"; mkdir -p "${d}"; DIRS+=("${d}")
done

for mode in fork thread subshell; do
    PIDS=()
    START=$(date +%s%N)

    for dir in "${DIRS[@]}"; do
        "${SCRIPT_DIR}/canaryfs" -${mode:0:1} -p 20 "${dir}" &   # 20 canaries × 10 dirs = 200
        PIDS+=($!)
    done

    sleep 3

    TRIGGER_START=$(date +%s%N)
    for dir in "${DIRS[@]}"; do
        for f in .env id_rsa credentials.json shadow.bak backup.sql; do
            [[ -f "${dir}/${f}" ]] && cat "${dir}/${f}" &>/dev/null &   # trigger all types
        done
    done
    wait
    TRIGGER_END=$(date +%s%N)

    sleep 2
    for pid in "${PIDS[@]}"; do kill "${pid}" 2>/dev/null; done
    END=$(date +%s%N)

    echo "Total runtime    : $(( (END - START) / 1000000 )) ms"
    echo "Trigger duration : $(( (TRIGGER_END - TRIGGER_START) / 1000000 )) ms"
    echo "Canaries watched : $(wc -l < /tmp/.canaryfs_registry 2>/dev/null)"

    for dir in "${DIRS[@]}"; do sudo "${SCRIPT_DIR}/canaryfs" -r "${dir}"; done
done
rm -rf "${BASE}"

What to look for in results

  • Fork vs Thread speed: Thread mode should start faster because creating a pthread is cheaper than forking a process.
  • Trigger time: How long it takes for the attacker's cat command to complete — should be near 0ms since the file is tiny.
  • Detection latency: The delay between the trigger and the ALERT appearing in the log. This depends on inotify event delivery speed.
  • Subshell under heavy load: Should handle 200 canaries fine since it's essentially the same as fork but detached.
